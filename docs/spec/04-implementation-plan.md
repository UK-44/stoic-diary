# 04. 実装計画 / マイルストーン

各マイルストーン（M）は独立して動作確認できる単位。**仕様承認後に M0 から着手**する。

## M0: 基盤整備
- [ ] Next 16 の App Router / Server Actions ガイドを `node_modules/next/dist/docs/` で確認
- [ ] Prisma スキーマを 02 の案で置き換え、`generator` を Prisma 7 仕様で確定
- [ ] `prisma migrate` で DB 反映、`lib/db.ts`（シングルトン）作成
- [ ] アプリ全体のレイアウト/ダークテーマの土台、ナビゲーション
- **受け入れ**: `npm run build` 成功、DB にテーブル作成、トップが表示される

## M1: 認証（Supabase Auth・メール認証）
- [ ] `@supabase/supabase-js` + `@supabase/ssr` 導入、`lib/supabase/{server,client}.ts`
- [ ] `/login`（メール＋パスワード）、`auth/callback`、サインアウト
- [ ] `middleware.ts` でセッション更新＋未認証リダイレクト（全ルート保護）
- [ ] `lib/auth.ts`（現在のユーザー取得・`User` upsert・所有者チェック）
- [ ] Supabase 側でオーナー 1 アカウントを発行、サインアップ無効化
- **受け入れ**: オーナーのみログイン可、未認証はブロック

## M2: マスタ管理
- [ ] コンポーネントマスタ CRUD（`/admin/components`、アーカイブ対応）
- [ ] フォーム構成マスタ CRUD（`/admin/forms`、版・並び・固定メッセージ文面）
- [ ] `prisma/seed.ts` で現行 Notion 構成（02 のシード表）を投入
- [ ] `lib/form-resolver.ts`（日付→有効版の解決）
- **受け入れ**: 現行構成の版が 1 つ存在し、部品の追加/並べ替え/アーカイブができる

## M3: 日記入力（コア機能）
- [ ] `/diary/[date]` 動的フォーム描画（型別フィールド: FixedMessage / BulletList / GroupedList）
- [ ] コア項目（目標 / 総合評価 4 段階）入力
- [ ] Server Action で `(userId,date)` upsert・値の保存・`formVersionId` スナップショット
- [ ] 日付ナビゲーション（前日/翌日/今日）
- **受け入れ**: 今日の日記を入力→保存→再読込で復元、保存成否が明示される

## M4: 一覧・閲覧
- [ ] トップ（`/`）にエントリ一覧（新しい順）、今日へのショートカット
- [ ] 過去日は記録された版（`formVersionId`）で当時の構成のまま表示（FR-8）
- **受け入れ**: 一覧から任意日へ遷移、過去構成が崩れず表示される

## M5: Notion 移行
- [ ] Notion「Markdown & CSV」エクスポートを受領（OQ-2）
- [ ] `scripts/import-notion.ts`（CSV→コア項目、md→コンポーネント値、冪等）
- [ ] `総合評価` 文字列→数値マップ確定（OQ-3）
- [ ] 実データで投入し、新アプリで過去日が正しく見えることを検証
- **受け入れ**: 過去エントリが移行され、アプリ上で閲覧できる（再実行しても重複しない）

## M6: 仕上げ
- [ ] バリデーション・エラーハンドリング・空状態
- [ ] スタイル調整（Notion 風のシンプルな見た目）
- [ ] デプロイ（Vercel 想定）と本番環境変数設定
- **受け入れ**: 日常運用に耐える状態

## 依存・順序
- M1 は OQ-1（ログイン方式）確定が前提。未確定なら M0→M2→M3 を先行し、認証は後付けも可。
- M5 は OQ-2/OQ-3 と実データ受領が前提。
- まず動かしたいなら **M0 → M2 → M3** を最短経路として優先する。
